A folder convention with structure that actually holds

AgentComms is a three-zone filesystem protocol. Each zone has a single rule. Everything else follows naturally.

No cloud dependency. No npm. No build step. A shell script creates your instance. The dashboard gives you live visibility. Agents work from plain Markdown — the same format they already read and write.

The repo is the installer. Your AgentComms instance lives wherever you want it — local, shared drive, or any path your agents can reach.

agents/*/inbox/

Routing signals only. Inbox files say "there's work for you in threads/." Small, fast, easy to scan. Nothing more.

threads/

All active work. Each task gets a folder: brief, context, status, Q&A, and result. One folder, full artifact trail.

archive/

Completed threads. Everything done moves here. The archive is never deleted. History is always intact.

What makes it different

Most agent coordination tools require infrastructure. AgentComms requires a folder and a convention.

📂

Filesystem-native

Every agent, every tool, every environment can read and write files. No integration work. No API keys. No platform lock-in.

📄

Brief-first discipline

No brief, no work. Every task starts with a written spec. The brief is the contract — between the requester, the agent, and the record.

🔍

Human-observable at every point

The dashboard shows who has work, what's in flight, and what's blocked — without interrupting any agent. Visibility without disruption.

📦

Zero-dependency server

The dashboard runs on Node.js built-ins only. No package.json. No node_modules. Works with Node ≥ 14, anywhere.

♻️

Persistent artifact trail

Q&A, briefs, results, and status all live in the thread folder. The archive is never deleted. You can reconstruct exactly what happened and why.

🧩

Any agent, any model

AgentComms is model-agnostic. OpenAI, Anthropic, open-source — if it can read and write files, it works with AgentComms.

How it works

Three steps. Every agent. Every task.

1
inbox/

Signal arrives

A routing signal lands in the agent's inbox. It points to a thread folder. One file, one pointer — nothing more.

2
threads/

Agent does the work

The agent reads the brief, asks questions in the thread, does the work, writes the result. The full artifact trail builds as it goes.

3
archive/

Thread is archived

Status updates to done. The thread moves to archive. Inbox clears. The next task begins. Nothing is lost.

Get started in one command

Clone the repo, run setup, and you have a working AgentComms instance with an example team and a live dashboard.

# Clone the installer git clone https://github.com/theProductPath/agentcomms.git # Move to where you want your AgentComms to live cd ~/Documents # Run setup bash ~/agentcomms/setup.sh # Start the dashboard at localhost:7843 bash AgentComms/dashboard/start.sh
